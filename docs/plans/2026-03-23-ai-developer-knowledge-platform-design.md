# AI 开发者平台专家库 — 产品设计文档

> 下载一个知识包，你的 Agent 就变成那个平台的资深开发者。

## 1. 产品定位

面向 AI 开发者（Vibe Coding 用户）的平台知识生态。

核心洞察：AI Agent（Claude Code、Cursor 等）已经能写代码、改代码、跑命令，但它们不了解特定平台的开发文档。开发者每次接入一个新平台，都要花大量时间让 Agent "学习" 文档，过程低效且浪费 token。

本平台解决的问题：把平台方的开发文档转化为高质量的「知识包」，让 Agent 即插即用地成为该平台的资深开发者。

### 三个层级对比

| 层级 | 做什么 | 例子 |
|------|--------|------|
| 文档搜索 | 用户搜 → 自己读 → 自己解决 | 传统文档站 |
| RAG 问答 | 用户问 → AI 找到文档 → 告诉你答案 | ChatGPT + 知识库 |
| **本平台** | **用户说需求 → Agent 找到知识 → 直接帮你干** | 知识包 + 现有 Agent |

### 目标用户

- C 端：用 AI Agent 开发软件的开发者（Vibe Coders）
- B 端（未来）：希望让更多 AI 开发者接入自己生态的平台方

## 2. 竞品分析

| 产品 | 定位 | 与本平台的区别 |
|------|------|---------------|
| Context7（Upstash）| 文档聚合 + MCP 检索 | 免费引流工具，无 B 端模式，MCP 持续连接不必要 |
| Mintlify | AI 原生文档站（$18.5M A 轮） | 面向人类阅读，不面向 Agent |
| Inkeep | Agent 构建平台 + Skill 生成 | Skill 是附属功能，核心是 Agent 平台 |
| skills.sh（Vercel）| Skill 包管理器 | 只管分发，不管内容质量 |
| DevDocs.io | 文档聚合 | 面向人类，无 AI 化 |

**市场空白**：没有人专门做「让 Agent 变成某平台专家」的高质量知识包。

## 3. 知识包结构

一个知识包 = 标准的 Agent Skill 目录，遵循 agentskills.io 规范：

```
douyin-minigame/
├── SKILL.md                    ← 专家大脑（元数据 + 工作流 + 决策指引）
└── references/                 ← 按主题分的深度指引
    ├── getting-started.md      ← 从零开始的完整流程
    ├── project-structure.md    ← 项目结构和配置说明
    ├── payment.md              ← 支付接入全流程
    ├── ad-monetization.md      ← 广告变现接入
    ├── social-features.md      ← 社交功能（分享、排行榜等）
    ├── review-submission.md    ← 提审上线流程和注意事项
    ├── debugging.md            ← 常见报错和排查指南
    └── api-reference/          ← API 按模块分组
        ├── login-api.md
        ├── payment-api.md
        ├── ad-api.md
        └── ...
```

### 三层渐进式披露

| 层 | 内容 | 加载时机 | Token 消耗 |
|----|------|---------|-----------|
| SKILL.md 元数据 | 名称、描述、触发条件 | Agent 启动时 | ~200 tokens |
| SKILL.md 正文 | 工作流总览、决策指引、踩坑清单 | Skill 被激活时 | ~800 tokens |
| references/*.md | 具体主题的完整指引 | Agent 做具体任务时按需加载 | ~500-1500 tokens/文件 |

对比读网页：一次查询约 1500-2500 tokens vs 读网页 5000-10000 tokens，节省 70-85%。

### SKILL.md 设计原则

SKILL.md 不只是一个「搜索入口」，而是一个完整的专家知识体系：

1. **流程总览** — 开发一个完整项目的步骤串联
2. **决策指引** — 要做 X 功能时该看哪个 reference
3. **踩坑清单** — 资深开发者才知道的注意事项
4. **调试指南** — 常见问题的排查方向

### 图片处理策略

原始文档中的图片不直接包含在知识包中，而是转化为 Agent 可执行的文字：

| 图片类型 | 转化方式 |
|---------|---------|
| 流程图/架构图 | 转成文字步骤或 Mermaid |
| 后台截图 | 转成操作路径指引 |
| 代码运行截图 | 转成实际代码块 |

## 4. 平台架构

```
┌─ 处理引擎（本地工具链）────────────────────────────┐
│  文档采集 → 清洗 → 图片转文字 → 内容重组 → 生成 Skill │
└───────────────────────┬────────────────────────────┘
                        ↓
┌─ 分发网站 ────────────────────────────────────────┐
│  知识包列表 → 平台介绍 → 详情页 → 下载/安装        │
└───────────────────────┬────────────────────────────┘
                        ↓
┌─ 开发者使用 ──────────────────────────────────────┐
│  下载知识包 → 放入项目 → Agent 自动识别 → 开始开发  │
└──────────────────────────────────────────────────┘
```

### 处理管线（6 个阶段）

1. **采集** — 获取原始文档（手动下载/爬取文档站）
2. **清洗** — 去除导航、页脚、重复内容，统一格式
3. **图片转文字** — 调用多模态 AI（Gemini）将图片转为文字描述
4. **内容重组** — 按主题聚合散落内容，补充工作流串联，标注常见坑
5. **生成 SKILL.md** — 基于 references 结构自动生成专家导航
6. **人工质量审查** — 通读检查 + 实际 Agent 测试

### 不使用 RAG 的理由

国内平台 ToC 开发文档的规模（100-300 页）完全可以用 well-organized markdown 覆盖。RAG 的额外 token 节省（相比 .md 约 50-60%）不足以抵消其开发成本和用户使用门槛。未来文档量级增大后可随时加入。

### 不使用 MCP 的理由

文档查询是无状态的一问一答，不需要 MCP 的持续连接。Skill + references 的按需加载已经足够高效，开发者端零依赖、零运维。

## 5. 技术栈

### 处理引擎

- Python 3.12+
- 文档采集：requests + BeautifulSoup
- 文档清洗：自写 Python 脚本
- 图片转文字：Gemini API（通过 Lumos 代理）
- 内容重组：Claude API 辅助 + 人工审阅
- SKILL.md 生成：Jinja2 模板引擎

### 分发网站

- Next.js（静态生成）
- 部署：Vercel（免费额度）
- 知识包托管：GitHub public repo
- 安装兼容：npx skills add / 直接下载 zip

## 6. MVP 范围

### 第一批内容：仅抖音小游戏

集中精力做好一个标杆知识包，验证核心价值。

### 做什么

| 优先级 | 内容 |
|--------|------|
| P0 | 处理管线脚本（6 个阶段） |
| P0 | 抖音小游戏知识包（完整） |
| P0 | 分发网站（首页 + 详情页 + 下载） |
| P1 | 快速开始教程 |

### 不做什么

- 用户注册/登录
- 在线上传后台
- 自动更新同步
- 计费系统
- RAG
- CLI 工具

### 验证标准

1. 用装了知识包的 Claude Code 做一个抖音小游戏 demo，全程不需要手动查文档
2. 对比没装知识包时的 token 消耗和完成速度，有明显差异
3. 至少 10 个外部开发者试用并给出反馈

## 7. 未来扩展方向

按优先级排列：

1. **更多平台知识包** — 微信小程序、支付宝小程序、Unity 等
2. **B 端自助接入** — 平台方自己上传文档，自动生成知识包
3. **社区贡献机制** — 开发者可以提交改进和补充
4. **自动更新** — 监听文档变更，自动重新生成知识包
5. **RAG 增强** — 当单个平台文档量超过 .md 可管理范围时引入
6. **使用数据分析** — 告诉 B 端「哪些文档被查最多、哪里知识不够」
