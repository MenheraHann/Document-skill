# 提审上线全流程

## 提审前必做检查清单

在提交审核前，逐项确认：

| 检查项 | 说明 | 必须 |
|--------|------|------|
| 侧边栏复访 | 已接入侧边栏能力 | 所有游戏 |
| 客服能力 | 已接入客服入口 | IAP 游戏 |
| 敏感词过滤 | 对用户输入做了过滤 | 有用户输入的游戏 |
| 域名 HTTPS | 所有域名用 `https://` 或 `wss://` | 所有游戏 |
| 基础信息 | 名称/头像/简介/软著全部审核通过 | 所有游戏 |
| 隐私政策 | 已设置隐私政策 | 所有游戏 |
| 适龄提示 | 已设置适龄提示 | 所有游戏 |
| 自审自查 | 已完成自审自查确认 | 所有游戏 |
| 包体大小 | 不超过 4MB（超出用分包加载）| 所有游戏 |
| 测试代码 | 移除 console.log 和测试代码 | 所有游戏 |
| 防沉迷 | 接入中宣部防沉迷实名认证系统 | 所有游戏 |

## 版本阶段说明

| 版本阶段 | 说明 |
|---------|------|
| 测试版本 | 开发完成上传到平台，扫码测试体验 |
| 审核版本 | 测试版本提交审核，等待平台审核 |
| 灰度版本 | 面向部分用户的线上版本（需先有线上版本）|
| 线上版本 | 面向全部用户的正式版本 |

## 提审流程

### 1. 上传游戏包

**普通小游戏**：在开发者工具中点击「上传」按钮，填写版本号和描述。

**Unity 小游戏**：通过 Unity 小游戏转换工具导出后，在开发者工具中上传。

> 最多支持 25 个测试版本并行，支持多通道测试。

### 2. 测试验证

上传后在开发者后台生成测试二维码，用抖音扫码测试：
- 验证核心玩法流程
- 验证支付/广告功能
- 验证在不同设备上的表现

### 3. 提交审核

1. 登录 [开发者控制台](https://developer.open-douyin.com/console)
2. 进入小游戏 → 版本管理
3. 选择已测试完成的版本 → 提交审核
4. 填写审核说明（描述游戏玩法、变现方式等）

### 4. 审核结果处理

| 结果 | 处理 |
|------|------|
| 审核通过 | 可选择全量发布或灰度发布 |
| 审核不通过 | 根据拒绝原因修改后重新提交 |

### 常见审核拒绝原因

1. **未接入必接能力** — 侧边栏/客服/敏感词未接入
2. **域名不合规** — 使用了 http 而非 https
3. **缺少软著** — 未提供软件著作权登记证书
4. **内容违规** — 游戏内容不符合平台规范
5. **功能不完整** — 游戏崩溃、核心功能无法使用
6. **包体过大** — 超过 4MB 未做分包

### 5. 发布上线

**全量发布**：所有用户可见，适合功能稳定的版本。

**灰度发布**：面向部分用户，适合大更新的渐进式发布。前提是已有线上版本。

## 发布后数据监控

发布后可在开发者后台查看：

| 数据类型 | 包含指标 |
|---------|---------|
| 总览数据 | DAU、新增用户、留存率 |
| 用户分析 | 用户画像、行为路径 |
| 性能分析 | 启动耗时、帧率、崩溃率 |

## 版本更新流程

1. 开发新版本 → 上传测试版本
2. 测试通过 → 提交审核
3. 审核通过 → 全量/灰度发布
4. （可选）配置版本更新提醒，通知用户更新

## 上线后运营推广

| 方式 | API / 操作 | 说明 |
|------|-----------|------|
| 搜索配置 | 开发者后台设置标签 | 优化搜索排名 |
| 直玩获客 | 平台自动推荐 | 推荐流中直接试玩 |
| 订阅消息 | `tt.requestSubscribeMessage` | 向用户发送游戏通知 |
| 关注抖音号 | `tt.openAwemeUserProfile` | 引导关注官方号 |
| 粉丝群聊 | 开发者后台配置 | 建立玩家社区 |

## 常见踩坑速查

| 问题 | 后果 | 解决 |
|------|------|------|
| 没接侧边栏复访 | 审核必挂 | 所有游戏上线前必须接入 |
| 没有软著就提审 | 无法通过审核 | 先申请软件著作权登记证书 |
| 域名用 http | 审核不通过 | 所有域名必须配置 https:// 或 wss:// |
| 内购游戏没接客服 | 审核被拒 | IAP 游戏必须接入客服能力 |
| 有用户输入没过滤 | 审核被拒 | 接入敏感词过滤 API |
| 激励视频顶层创建 | 静默失败 | 在游戏初始化完成后创建实例 |
| 包体超 4MB | 无法上传 | 使用分包加载 |
| getUserInfo 已废弃 | 接口报错 | 改用 tt.getUserProfile |
| iOS 端用虚拟支付 | 审核被拒 | iOS 必须走平台内购通道 |
| 支付回调客户端验证 | 审核被拒 | 必须做服务端签名验证 |

## 附录 A：提审自查清单详解

### 1. 侧边栏复访能力

- **检查方法**：确认已调用 `tt.showFavoriteGuide()` 引导用户添加到侧边栏
- **判定规则**：游戏内必须有至少一处触发引导的时机（如首次进入、通关后）
- **官方文档**：开放平台 → 小游戏 → 能力接入 → 侧边栏复访

### 2. 软件著作权登记证书

- **检查方法**：在开发者控制台 → 基础信息中确认软著已上传且审核通过
- **判定规则**：必须是国家版权局颁发的正式登记证书，电子软著可准入但不可用于备案
- **官方文档**：开放平台 → 小游戏 → 基础信息管理

### 3. 域名 HTTPS/WSS 配置

- **检查方法**：检查所有 `tt.request`、`tt.connectSocket` 调用的 URL 前缀
- **判定规则**：所有网络请求域名必须以 `https://` 或 `wss://` 开头，且已在开发者后台配置白名单
- **官方文档**：开放平台 → 小游戏 → 开发 → 网络

### 4. 客服能力（IAP 游戏）

- **检查方法**：确认已调用 `tt.openCustomerServiceConversation()` 或在游戏内嵌入客服入口
- **判定规则**：仅 IAP 游戏必须接入；纯 IAA 游戏不强制
- **官方文档**：开放平台 → 小游戏 → 能力接入 → 客服

### 5. 敏感词过滤（有用户输入）

- **检查方法**：确认所有用户输入（昵称/聊天/评论）经过服务端内容安全检测 API
- **判定规则**：仅有用户自定义输入的游戏必须接入；无输入不强制
- **官方文档**：开放平台 → 内容安全 → 文本检测 API

### 6. 支付回调服务端验证

- **检查方法**：确认支付成功回调在服务端做签名验证，而非仅在客户端处理
- **判定规则**：客户端验证会被拒审，必须走服务端 → 平台 API 验证流程
- **官方文档**：开放平台 → 小游戏 → 支付 → 服务端验证

### 7. 包体大小 ≤ 4MB

- **检查方法**：在开发者工具中查看构建产物大小，或检查上传时的包体大小提示
- **判定规则**：主包 ≤ 4MB；超出部分必须使用分包加载
- **官方文档**：开放平台 → 小游戏 → 开发 → 分包加载

### 8. 未使用已废弃 API

- **检查方法**：全局搜索 `getUserInfo`、`tt.login`（旧版）等已废弃接口
- **判定规则**：`getUserInfo` 必须替换为 `tt.getUserProfile`
- **官方文档**：开放平台 → 小游戏 → API → 用户信息

### 9. iOS 虚拟支付走内购通道

- **检查方法**：确认 iOS 端调用 `tt.requestGamePayment` 时使用平台内购参数
- **判定规则**：iOS 端不支持直接虚拟支付，必须走平台内购通道
- **官方文档**：开放平台 → 小游戏 → 支付 → iOS 支付

### 10. 真机预览无报错

- **检查方法**：在开发者工具中生成预览二维码，用抖音扫码在真机上测试全部核心功能
- **判定规则**：无崩溃、无白屏、核心功能（支付/广告/分享）正常运行
- **官方文档**：开放平台 → 小游戏 → 开发 → 调试
